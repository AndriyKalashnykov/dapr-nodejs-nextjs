# Migration plan: Knex.js → Kysely

> **Status: deferred (2026-05-02).** Filed as a backlog item in `CLAUDE.md`. This document is the plan to execute when the work is taken on.

## Decision: Kysely + keep `pg` driver

Earlier research considered three typed query builders (Kysely, Drizzle ORM, Prisma) and two drivers (`pg`, `postgres.js`). Verdict: **Kysely on top of `pg`**, with a `postgres.js` swap as an optional Phase 6.

### Why Kysely over Drizzle / Prisma for this codebase

| Dimension | **Kysely** | Drizzle | Prisma |
|---|---|---|---|
| Type safety origin | Hand-written `Database` interface OR generated by `kysely-codegen` (introspects live DB) | Hand-written schema in TS DSL | Generated client from `schema.prisma` DSL |
| Migrations | `Migrator` with TS files containing `up`/`down` (semantically identical to Knex's). Tracks state in its own `kysely_migration` table — does NOT conflict with `knex_migrations` | `drizzle-kit generate` produces SQL diffs from the TS schema; no `up`/`down` parity with Knex | Prisma Migrate owns the schema; rewrite required |
| Transaction API | `db.transaction().execute(async trx => …)` (auto-commit) **OR `ControlledTransaction` (manual `commit`/`rollback`)** since 0.27 | Callback-only auto-commit; no first-class manual control | `prisma.$transaction([...])` or interactive callback |
| Driver flexibility | `pg` (`PostgresDialect`) **and** `postgres.js` (`PostgresJSDialect`) | Both supported via separate adapters | Prisma owns the connection (Rust binary or new TS engine) |
| Maintenance health (2026-Q1/Q2) | Active; Koskimas + community; ~14k stars; minor releases monthly | Very active but high churn; several breaking renames in `drizzle-orm` 0.30→0.40 | Active but Prisma 6 brought engine swap |
| Soft-delete pattern | `.where('deleted_at', 'is', null)` — direct fit | Same — direct fit | Awkward; needs middleware/extension |
| `snake_case` ↔ `camelCase` | `CamelCasePlugin` ships in core; opt-in per-instance | Per-column boilerplate in schema | `@map`/`@@map` requires the rewrite |
| `asModel()` interaction | Plays nicely; delete `asModel` once `CamelCasePlugin` is on | Same | Generated types replace `asModel` entirely (rewrite) |

### The three reasons that decide it

1. **Migration files survive a 1-line rename.** Kysely's `up(db)`/`down(db)` semantics match Knex's exactly. The single existing migration (`20250205181503_todo-table.ts`) converts in minutes — `Knex` → `Kysely<any>`, `knex.raw(…)` → `` sql`…`.execute(db) ``. Drizzle would force a schema-DSL rewrite; Prisma a paradigm shift.
2. **`ControlledTransaction` matches Knex's manual `trx.commit()`/`trx.rollback()` exactly.** `services/todo.ts` uses explicit commit *after* `PubSub.publish()` so a publish failure rolls back the DB write. Kysely is the only one of the three that supports this pattern without inverting the control flow.
3. **Driver-agnostic.** Adopt Kysely on top of the existing `pg` pool (zero connection-layer risk — same SSL config, same pool events, same Dapr-secretstore wiring). Defer the `postgres.js` decision to Phase 6.

### On `pg` vs `postgres.js`

Keep `pg` for this migration. `postgres.js` is faster (~10–25% on small queries, more on prepared statements) and has cleaner async ergonomics, but: (a) it's a separate change with its own risk surface (different connection-error handling, no built-in `Pool` events, different SSL config shape, different `NOTIFY`/`LISTEN` semantics); (b) the upgrade backlog flags `pg` as "monitor", not "migrate"; (c) `buildServiceContext()` and the schema-search-path config in the SDK were validated against `pg` and would need re-validation. Phase 6, optional, deferrable.

## Phased implementation plan

### Phase 1 — Add Kysely alongside Knex (toolchain only)

- **Goal**: both `context.db` (Knex) and `context.kysely` (Kysely) available; nothing migrated.
- **Files**:
  - `app/backend-ts/package.json` — add `kysely`, `kysely-ctl` (devDep), `kysely-codegen` (devDep). Keep `knex`, `pg`.
  - `packages/@sos/sdk/src/db/db.ts` — add `kyselyClient(serviceSecrets, config)` that constructs `new Kysely({ dialect: new PostgresDialect({ pool: new Pool(...) }), plugins: [new CamelCasePlugin()] })` reusing the **same connection params** Knex uses.
  - `packages/@sos/sdk/src/types.ts` — extend `Context<K>` with `kysely: Kysely<Database>` alongside `db: Knex`.
  - `packages/@sos/sdk/src/context.ts` — populate both in `buildServiceContext()`.
  - `app/backend-ts/src/db/schema.ts` (new) — generated `Database` interface from `kysely-codegen --url postgres://… --schema backend_ts`.
- **Order**: install → generate types → wire SDK → expose on `Context` → smoke compile.
- **Risk**: connection pool double-up (now 2× max=10). **Mitigation**: lower each side's `max` to 5 during transition, OR share a single `pg.Pool` between both clients.
- **Verification**: `make test` and `make integration-test` still pass unchanged. Smoke: `make up -d && curl …/api/v1/todos` returns rows.
- **Effort**: ~3h.

### Phase 2 — Migrate `todo` model + service end-to-end as proof point

- **Goal**: `models/todo.ts` and `services/todo.ts` use `context.kysely`; Knex code path removed for this slice.
- **Files**:
  - `app/backend-ts/src/models/todo.ts` — rewrite all 5 functions. Key conversions:
    - `context.db<TodoDb>(TABLE).where('deleted_at', null).count(...)` → `context.kysely.selectFrom('todo').where('deletedAt', 'is', null).select(({ fn }) => fn.countAll<number>().as('count')).executeTakeFirst()`
    - `trx<TodoDb>(TABLE).insert({...}).returning('*')` → `trx.insertInto('todo').values({...}).returningAll().executeTakeFirstOrThrow()`
    - With `CamelCasePlugin`, **drop `asModel()`** — return rows directly. Confirm with side-by-side log of one `getAll` response in dev before deleting.
  - `app/backend-ts/src/services/todo.ts` — replace `await context.db.transaction()` + manual `trx.commit/rollback` with Kysely's `ControlledTransaction`:
    ```ts
    const trx = await context.kysely.startTransaction().execute();
    try { … await trx.commit().execute(); } catch (e) { await trx.rollback().execute(); throw e; }
    ```
  - `app/backend-ts/src/types.ts` — `TodoDb` becomes `Selectable<Database['todo']>`; `CreateTodoModel`/`UpdateTodoModel` use `Insertable`/`Updateable`.
  - `app/backend-ts/src/lib/test/models/todo.ts` — update factory to use Kysely.
- **Order**: model first (with both `db` and `kysely` available); compile + unit tests; service second; integration tests last.
- **Risk**: `CamelCasePlugin` only renames in `select`/`insert`/`update` — raw SQL fragments and `orderBy('created_at', …)` from `Api.QueryParams` need either the plugin's auto-translate (it does handle `orderBy` column refs) or explicit camelCase. The integration test asserts `orderBy: 'created_at'` in the **response** — that's the API contract, untouched. **Mitigation**: keep query-param column names as snake_case at the API boundary; let the plugin handle translation only inside Kysely's column refs.
- **Verification**:
  - `models/todo.test.ts`, `services/todo.test.ts` (unit) pass with mocks updated.
  - `handlers/api/todo.integration.test.ts`, `services/todo.cache.integration.test.ts`, `services/todo.pubsub.integration.test.ts` pass **unchanged** (API contract is the gate).
  - New `models/todo.kysely.integration.test.ts` exercising the count/orderBy/limit/offset path explicitly with a generated SQL assertion via `db.compile()`.
  - `EXPLAIN` parity check: log Kysely-generated SQL for `getAll` and `getById`, run `EXPLAIN` against both Knex and Kysely versions in `make psql`, confirm same plan.
  - Manual smoke: full `make up -d` + `curl` round-trip.
- **Effort**: ~6h.

### Phase 3 — Migrate the rest

- **Goal**: every Knex query-builder call site converted.
- **Files**: this codebase only has `models/todo.ts` and `services/todo.ts` as Knex consumers (`services/user.ts` doesn't touch the DB). So Phase 3 is **a no-op for current code** but should include a grep guard:
  - `grep -rn "context\.db\|Knex\." app/backend-ts/src` must return zero.
- **Risk**: scaffolds in `scaffolds/` may still emit Knex code. **Mitigation**: update `scaffolds/` templates and the "Adding a New Service" doc.
- **Verification**: `make ci` green.
- **Effort**: ~0.5h.

### Phase 4 — Cut over migration tooling

- **Goal**: `pnpm run knex -- migrate:latest` replaced by `pnpm run migrate` running Kysely's migrator.
- **Files**:
  - `app/backend-ts/src/db/migrations/20250205181503_todo-table.ts` — convert: `import type { Kysely } from 'kysely'; import { sql } from 'kysely'; export async function up(db: Kysely<any>) { await sql\`CREATE TABLE …\`.execute(db); }`. The CREATE TABLE SQL itself is unchanged.
  - `app/backend-ts/src/db/migrate.ts` (new) — `Migrator` instance pointing at `./migrations`, programmatic runner.
  - `app/backend-ts/src/db/knexfile.ts` — **delete** (Phase 5).
  - `app/backend-ts/package.json` — `"dev"` and `"test:integration"` scripts: replace `pnpm run knex -- migrate:latest` with `tsx ./src/db/migrate.ts up`.
  - `Makefile` — `migrate` target updated.
- **Decision on existing migrations**: with one migration file and a small dataset (truncated between tests, recreated from compose for prod), the cleanest path is: (a) drop `knex_migrations` row, (b) convert the file, (c) let Kysely re-record it in its own `kysely_migration` table on the first run **against an empty schema** — for `backend_ts_test` the schema is recreated anyway; for any deployed `backend_ts` use a one-time SQL `INSERT INTO kysely_migration (name, timestamp) VALUES ('20250205181503_todo-table', NOW())` to mark it as already-applied.
- **Risk**: deployed Azure Postgres has the migration recorded only in `knex_migrations`; if Kysely re-runs it the `CREATE TABLE` will fail. **Mitigation**: ship the seed-row SQL as a one-time pre-deploy step in the same PR; document in PR description.
- **Verification**: `make integration-test` (drops + recreates schema) passes. `make migrate` against a fresh DB creates the `todo` table. `make migrate` against a populated DB is a no-op.
- **Effort**: ~3h.

### Phase 5 — Remove Knex + cleanup

- **Goal**: zero Knex references.
- **Files**:
  - `app/backend-ts/package.json` — remove `knex` dep, remove `"knex"` script.
  - `packages/@sos/sdk/src/db/db.ts` — delete `buildKnexConfig`, `knex`; keep only the Kysely client builder.
  - `packages/@sos/sdk/src/types.ts` — drop `db: Knex` from `Context`, rename `kysely` → `db`.
  - `packages/@sos/sdk/src/context.ts` — drop the Knex assignment.
  - `app/backend-ts/src/db/knexfile.ts` — delete.
  - `app/backend-ts/src/models/todo.ts`, `services/todo.ts` — rename `context.kysely` back to `context.db`.
  - `CLAUDE.md` "Database" and the `knex` line in "Backend Layering" — update (copy below).
  - `scaffolds/` templates — remove Knex references.
  - **Upgrade Backlog** — strike the `pg` solo-maintainer item if Phase 6 is taken; otherwise update the note.
- **Verification**: `grep -rn knex app/ packages/ scaffolds/` returns zero. Full `make ci` + `make integration-test` + `make e2e` green.
- **Effort**: ~2h.

#### New `CLAUDE.md` "Database" section copy

```
### Database

- PostgreSQL 18 with Kysely (typed query builder) over the `pg` driver
- Schema per service: `backend_ts` (prod), `backend_ts_test` (integration tests) — the `_test` suffix is auto-appended when `NODE_ENV=test`
- `Database` types are generated from the live schema via `kysely-codegen` into `src/db/schema.ts`; regenerate after every migration
- Migrations live in `app/backend-ts/src/db/migrations/` as Kysely `up`/`down` files; runner is `pnpm run migrate` (also wired into `make migrate`)
- DB credentials come from Dapr secretstore (not env vars) — see `buildServiceContext()` in SDK
- `CamelCasePlugin` translates `snake_case` columns to `camelCase` model fields automatically; do not hand-roll `asModel()` mappers
- Transactions use Kysely's `ControlledTransaction` so services can interleave DB writes with `PubSub.publish` and explicit `trx.commit()` / `trx.rollback()`
```

### Phase 6 (optional) — Switch driver `pg` → `postgres.js`

- **Goal**: replace `PostgresDialect` with `PostgresJSDialect`.
- **Files**: `packages/@sos/sdk/src/db/db.ts` (only); `app/backend-ts/package.json` (swap `pg` for `postgres`).
- **Risk**: SSL-config shape, connection-error event handling, schema search_path setup all differ. **Recommendation**: defer; revisit only if profiling shows pool/driver overhead is material.
- **Effort**: ~4h.

## Flag-day items (not incremental)

- **Phase 4 cutover**: the `migrations` directory can be owned by exactly one tool at a time on a given DB. The conversion + the `kysely_migration` seed row must land in a single PR / single deploy.
- **Phase 5 type rename** (`context.kysely` → `context.db`): single PR, all call sites at once. Trivial because `models/todo.ts` is the only consumer.

## Backlog items if phases are deferred mid-migration

- If only Phases 1-2 land: backlog "Finish Kysely migration: convert remaining models (none today, but scaffolds still emit Knex), then cut over migrator." Note the temporary 2× pool overhead.
- If Phase 4 deferred: backlog "Cut migrator from Knex to Kysely; one-time `kysely_migration` seed row required against deployed DBs." Until then `pnpm run knex -- migrate:make` still works for new migrations and is the documented path.
- If Phase 6 deferred: keep current Upgrade-Backlog item "pg solo-maintainer; monitor for fork." Kysely's dialect swap is ~10 lines if/when needed.

## Revisit triggers

Pull this plan back off the shelf when any of the following is true:

- Knex maintenance stalls (>6 months without a release while a CVE lands).
- A second model gets added to the backend (right now `todo` is the only consumer; the cost of staying on Knex grows with each new model).
- Type-safety friction with Knex starts producing real bugs that this migration would have prevented (track in backlog notes).
- `pg` solo-maintainer status changes adversely (separate backlog item) — at which point the natural pairing is `postgres.js` + Kysely, executed as Phases 1-6 in one pass.
